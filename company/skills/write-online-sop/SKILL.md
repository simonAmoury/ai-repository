# 上线 SOP 编写规范

## 用途

统一「上线 SOP」文档的结构、粒度与写法，确保每次发布文档都覆盖：外部配置变更、上线顺序、上线后自检、回滚方案四大要素，且每一条外部改动都可被运维照着执行、可判断「不处理的后果」。

## 何时使用

- 用户明确要求写 / 补充「上线 SOP」「发布文档」「上线清单」「变更说明」。
- 一次功能迭代收尾，需要沉淀部署步骤。
- **随手补充**：当你在改代码时引入了外部平台改动（Apollo / 飞书 / DDL / 三方接口），应立即把该条目补进对应 SOP —— 此时也套用本规范的条目格式。

## 核心原则

1. **改动即记录**：外部改动发生的当次就写进 SOP，不攒到最后回忆整理。
2. **可执行**：每条 Apollo 配置写清 服务 / appid / env / cluster / key / value，运维可直接照着定位并填值。
3. **无变更也要写明**：DDL 无变更、无需回滚等，也显式写「无」，而不是留空让人猜。

## 文档必备结构

按以下四节组织（小节可依实际增减，但四大块不可缺）：

1. **一、上线前需处理项**：Apollo 配置表、外部平台权限、数据处理、依赖升级回归。
2. **二、上线顺序**：服务部署先后及依赖原因。
3. **三、上线后自检**：勾选式验证清单，覆盖核心链路。
4. **四、回滚**：分「应用 / Apollo / 数据库 / 其他」列回滚动作。

完整骨架见 `resources/online-sop-template.md`，生成时以它为起点填充。

## 条目写法规范

### Apollo 配置表（必备列）

固定 6 列，只写这些，不加「处理 / 不处理的后果」等其它列：

| 列 | 要求 |
|----|------|
| 服务 | 服务名，如 `pac-bop-admin`、`pac-foreign` |
| appid | Apollo appid，按「服务坐标对照表」填 |
| env | Apollo 环境，如生产 `PRO-HWGZ` |
| cluster | Apollo 集群，按「服务坐标对照表」填 |
| key | 完整配置键，如 `af.apply.approve.card.cache.expire.days` |
| value | 该环境下要填的值 |

> 结尾统一提醒「配置改完在 Apollo **点发布**」。

#### 服务坐标对照表（生产环境 PRO-HWGZ）

按服务名查 appid / cluster，填入上表；其它环境的坐标以实际 Apollo 为准。

| 服务 | appid | env | cluster |
|------|-------|-----|---------|
| pac-foreign | pac-platform | PRO-HWGZ | pac-platform-cluster |
| pac-bop-admin | pac-bop | PRO-HWGZ | default |

### 外部平台 / 权限条目

写清：需要谁、在哪个平台、开通什么权限、不开通的报错表现、是否上线前置项。

### 数据处理（DDL / 数据订正）

- 有 DDL：贴出 SQL，标注执行库、执行时机、是否可回滚。
- 仅测试库操作：**显式标注「⚠️ 线上库不执行」**。
- 无 DDL：在回滚节写明「无 DDL 变更，无需回滚」。

### 上线后自检

用 `- [ ]` 勾选项，每条对应一个可观测结果（页面表现 / 卡片行为 / 接口返回），不写空泛的「验证功能正常」。

### 回滚表（必备行）

| 项 | 内容 |
|----|------|
| 应用 | 回滚到上一稳定镜像的动作 |
| Apollo | 各新增 key 是否需删除 / 保留 |
| 数据库 | 有无 DDL 需回滚 |
| 其他 | SDK / 三方约定等随镜像回滚说明 |

## 输出约定

- 用中文、Markdown 表格与勾选清单。
- 文件命名 `上线SOP.md`，放对应 spec / 迭代目录下。
- 生成后逐节对照本规范自查：四大块齐全、每条外部改动含「不处理的后果」、可选项已标注。
